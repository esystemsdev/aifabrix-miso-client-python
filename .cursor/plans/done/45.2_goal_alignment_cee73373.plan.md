## Goal
Implement strict token-contract behavior in `aifabrix-miso-client-python` by:
- removing residual alias/fallback parsing in strict auth paths,
- preserving explicit session/device refresh boundary,
- aligning tests and docs with canonical `accessToken`-first semantics,
- validating all changes through required repo gates before closure.

Result expected from this plan: strict auth paths in Python SDK no longer rely on alias/fallback parsing, explicit refresh boundaries remain intact, docs/tests are aligned to canonical behavior, and closure evidence is complete.

Reference context: [source cross-repo goal-alignment plan](/workspace/aifabrix-miso/.cursor/plans/192.2_goal_alignment_85d5f32e.plan.md).

## Goal Completion Criteria
The Goal is considered fully achieved only when all four conditions below are true at the same time:
- **G1 Parsing strictness**: no residual alias/fallback parsing remains in strict auth paths.
- **G2 Refresh boundary**: session and device refresh flows remain explicit and non-ambiguous.

- **G4 Validation closure**: required repo gates pass in order with evidence from `.temp/validation/*`.

## Contract Decisions for 45.2
- Canonical wire token field for strict path: `accessToken`.
- No ambiguous generic refresh entrypoint in API surface.
- Strict refresh manager path should not accept fallback payload/field aliases.
- For non-refresh auth response models, enforce canonical strict contract behavior and remove legacy alias acceptance.

## Allowed / Forbidden Matrix
- **Allowed**
  - Explicit refresh APIs only: `refresh_session_token()` and `refresh_device_code_token(refresh_token)`.
  - Canonical strict token field usage: `accessToken`.
  - Canonical contract behavior only (no backward-compatibility parsing paths).
- **Forbidden**
  - Reintroducing ambiguous generic refresh surface in public SDK API.
  - Fallback parsing in strict refresh manager paths (`flat` vs `nested` ambiguity, alias-only token acceptance).
  - Silent contract drift between implementation, tests, and docs/changelog.

## Token Storage Matrix (Python SDK Handoff)
- `refresh token`
  - Allowed location: server-side only.
  - Forbidden location: browser-readable payloads, `localStorage`, `sessionStorage`.
  - Python SDK runtime note: no browser runtime; code/docs/tests must not imply browser exposure.
- `user access token`
  - Allowed location: HttpOnly cookie semantics for browser flow (controller-owned); Python SDK may consume token strings in non-browser contexts.
  - Forbidden location: browser JS-managed persistent storage patterns in SDK guidance.
  - Python SDK runtime note: enforce contract language and parsing boundaries; do not reintroduce browser storage assumptions.
- `M2M client token`
  - Allowed location: runtime memory.
  - Forbidden location: persistent browser storage.
  - Python SDK runtime note: keep in-memory token handling patterns; no persistence guidance that contradicts source contract.

## Known Baseline Contradictions
- Residual alias acceptance in `auth_types.py` (`token` alias for `accessToken`) weakens strict canonical parsing intent.
- Any docs/examples that imply ambiguous refresh entrypoints or fallback token-field parsing are contract drift and must be removed.
- If additional contradictions are discovered during execution, they must be either remediated in this plan scope or the plan must be marked blocked/incomplete.

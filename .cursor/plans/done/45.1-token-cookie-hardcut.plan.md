# Plan 45.1 — Token-Cookie Hardcut (miso-client-python)

## Goal
Implement the `miso-client-python` hard-cut remediation scope from [`/workspace/aifabrix-miso/.cursor/plans/192.1-token-cookie-hardcut.plan.md`](/workspace/aifabrix-miso/.cursor/plans/192.1-token-cookie-hardcut.plan.md) and fallback findings report:

- enforce strict cookie/session vs device refresh boundaries,
- remove compatibility fallbacks in strict refresh paths,
- keep M2M `x-client-token` behavior unchanged.

## Hard-Cut Remediation Context

Based on report [`/workspace/aifabrix-miso/.temp/01-sdk-hardcut-fallback-report.md`](/workspace/aifabrix-miso/.temp/01-sdk-hardcut-fallback-report.md), the Python SDK must eliminate fallback behavior that weakens hard-cut enforcement:

- no multi-shape response fallback in strict refresh-manager paths;
- no token alias fallback (`token` vs `accessToken`) in strict paths unless explicitly mandated;
- no ambiguous refresh entry points that blend session and device semantics.

## Acceptance Criteria (No-Fallback Hard-Cut)
- Browser/session refresh strict path uses one canonical contract shape per endpoint.
- Device refresh keeps explicit request-body `refreshToken` contract and remains separate.
- Hard-cut execution path does not rely on response-shape fallback (`nested` vs `flat`).
- Hard-cut execution path does not rely on token-field alias fallback unless explicitly mandated by final contract.
- Ambiguous refresh API surface is removed/deprecated from strict flow expectations in code and tests.
- Regression coverage passes for:
  - session refresh (bodyless),
  - device refresh (`refreshToken` body),
  - unauthorized/expired refresh behavior,
  - M2M non-regression.

## Contract Mapping (Old -> New for Python SDK)
- Session refresh semantics:
  - Old: `/api/v1/auth/refresh` may be treated as body-token refresh path in some helper usage.
  - New: session-oriented behavior does not depend on request-body `refreshToken`, and strict paths avoid fallback response-shape aliasing.
- Device refresh semantics:
  - Old: `/api/v1/auth/login/device/refresh` with body `refreshToken`.
  - New: unchanged and explicit, with a single canonical response contract consumed by strict paths.
- M2M semantics:
  - Old: `x-client-token` service-to-service flow isolated from user session flow.
  - New: unchanged and explicitly guarded.

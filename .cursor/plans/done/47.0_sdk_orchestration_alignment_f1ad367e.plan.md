## Scope
### In scope
- Release/docs alignment for Python SDK as required by 442.0 handoff pack C.
- Request-driven runtime parity for Python SDK in shared contract areas:
  - deterministic `401` recovery flow with single-flight behavior under concurrency;
  - deterministic bounded handling for `403/422` (no restore/refresh loop path);
  - strict refresh API boundary (`refresh_session_token()` vs `refresh_device_code_token(refresh_token)`).
- Explicit scope boundary documentation: browser activity listener orchestration is TypeScript/browser SDK scope, not Python runtime scope for this rollout line.
- Cross-SDK migration terminology synchronization across Python SDK docs/changelog.
- Validation evidence for touched Python/docs paths.

### Out of scope
- Implementing browser activity orchestration runtime behavior in Python SDK.
- Introducing new Python runtime mandatory knobs/config fields.
- Unapproved Python runtime behavior changes not backed by verified mismatch and explicit approval.

## Fixed Decisions (from 442.0)
- Python SDK does not implement browser activity-listener orchestration in this rollout line.
- Approved runtime parity type is request-driven `Contract/runtime parity` only.
- Deliverables include request-driven runtime parity evidence plus release/docs alignment and terminology synchronization.
- Cross-SDK guidance must be consistent with TypeScript SDK final behavior.
- No new Python runtime mandatory knobs are introduced.

## Approved Runtime Parity Block
### Approved parity type
`Contract/runtime parity` (request-driven only).

### Approved invariants for this plan line
- Concurrent `401` request paths converge to one effective recovery attempt per user/token context (single-flight behavior).
- `401` recovery remains bounded/deterministic and does not create unbounded retry loops.
- `403` and `422` do not enter restore/refresh loops and remain deterministic.
- Refresh contract boundary remains explicit and strict:
  - cookie/session refresh via `refresh_session_token()` (`POST /api/v1/auth/refresh`, no body `refreshToken`);
  - device refresh-token flow via `refresh_device_code_token(refresh_token)` (`POST /api/v1/auth/login/device/refresh`).

### Explicit non-goals
- No browser activity trigger orchestration in Python runtime (`activity`, `visibilitychange`, `online` listeners remain TypeScript/browser scope).
- No new mandatory runtime config knobs mirroring browser DataClient listener controls.
